# 审查检查清单

逐原则的具体检查项。每项必须有 diff 证据支撑，不得凭印象判定。

## P1 — Think Before Coding

| # | 检查项 | 失败信号 |
|---|--------|---------|
| 1.1 | 变更是否基于验证过的前提 | 代码中出现"应该是"、"大概是"风格的 TODO/注释 |
| 1.2 | 方法选择是否经过确认 | 存在更简方案但未说明为何选当前方案 |
| 1.3 | 不确定处是否标注 | 有模糊逻辑但无注释说明假设 |
| 1.4 | 有没有该问没问就做的决策 | 改动引入了新的接口契约/数据格式但无讨论记录 |
| 1.5 | 声明性证据是否被当作运行时验证 | 从配置文件/文档/注释互证得出结论，但未验证实际存储内容或运行时调用 |

## P2 — Uncertainty Honesty（不确定性诚实）

| # | 检查项 | 失败信号（可从代码模式检测） |
|---|--------|--------------------------|
| 2.1 | 外部输入有存在性/类型守卫 | 对 API 返回值直接 `.field` 访问无 `?.` 或 `if` 守卫；对 dict 直接 `[key]` 无 `.get()`/`in` 检查 |
| 2.2 | 运行时可变值未伪装为编译期常量 | URL、超时值、schema 版本、第三方格式假设被硬编码为 `const`/全局变量，且无版本注释或过期标记 |
| 2.3 | 概率性结果有置信度/fallback | LLM 输出、模糊匹配、启发式判断的结果直接进入分支逻辑，无置信度阈值或 fallback 路径 |
| 2.4 | 时间依赖被显式处理 | 缓存无 TTL、Token 无过期检查、"最新数据"无时间戳验证 |

## P3 — Simplicity First

| # | 检查项 | 失败信号 |
|---|--------|---------|
| 3.1 | 没有未被要求的功能 | 新增了用户没提到的 feature flag / 配置项 / 抽象层 |
| 3.2 | 代码行数合理 | 200 行能 50 行解决的实现 |
| 3.3 | 没有不可能场景的错误处理 | try/except 包裹了框架保证不会抛出的代码路径 |
| 3.4 | 资深工程师不会说过度设计 | 三行相似代码被抽象为带泛型参数的工厂函数 |

## P4 — Surgical Changes

| # | 检查项 | 失败信号 |
|---|--------|---------|
| 4.1 | 每行 diff 可追溯到用户请求 | diff 中包含对相邻代码的格式调整/注释修改 |
| 4.2 | 匹配现有代码风格 | 新代码用了不同的命名规范/缩进/导入风格 |
| 4.3 | 自己产生的孤儿已清理 | 改动使某个 import/变量/函数不再使用但未删除 |
| 4.4 | 没有删改不属于此任务的代码 | 重构了"顺便发现"的技术债 |

## P5 — Goal-Driven Execution

| # | 检查项 | 失败信号 |
|---|--------|---------|
| 5.1 | 有可验证的成功标准 | 改动没有对应的测试运行 / 手动验证记录 |
| 5.2 | 多步任务有验证计划 | 多个文件的改动缺少中间验证步骤 |
| 5.3 | 验证实际执行了 | 声称"测试通过"但没有输出证据 |
| 5.4 | 回归检查已做 | 只验证新功能，未检查是否破坏现有行为 |

## P6 — SSOT-First Design

| # | 检查项 | 失败信号 |
|---|--------|---------|
| 6.1 | 无跨文件信息重复 | 同一规则/配置/说明在两个文件中各写了一份 |
| 6.2 | 业务数据从 SSOT 动态加载 | 硬编码了应从配置文件读取的常量 |
| 6.3 | SSOT 路径声明为模块级常量 | SSOT 文件路径散落在多个函数内部 |
| 6.4 | 变更的外部消费者已同步 | 改动涉及共享接口（函数签名/返回值/数据格式）或共享路径（文件重命名/删除/移动），但 diff 未包含依赖方的对应更新 |
| 6.5 | 隐式契约变更已识别 | 接口签名/文件名未变但语义变了（返回值含义、副作用、前置条件、文件内容结构），且存在外部依赖者 |

## 判定权限分级

❌ 判定分为两类，门槛不同：

### 技术否决

**定义**：代码有明确的逻辑/语法/契约错误（API 参数错误、类型不匹配、死循环、资源泄漏等）。

**门槛**：`file:line` + 代码片段 + 替代方案。与现有要求一致。

### 判断否决

**定义**：审查者认为原方案的设计决策有误——即推翻的不是"怎么写"而是"该不该做"。

典型信号：判定措辞含"不需要"、"过度"、"多余"、"应该直接"等对方案必要性的否定。

**门槛**：在技术否决的基础上，额外要求**反例验证步骤**：

| 字段 | 要求 |
|------|------|
| 步骤 | 30 秒内可执行（一条 grep/API 调用/文件读取） |
| 可证伪 | 结果能区分"原方案正确"和"审查结论正确" |
| 声明预期 | 写出"若 X 则原方案成立；若 Y 则审查结论成立" |

### 降级规则

附不出满足上述三条件的反例验证步骤时：

- ❌ 降级为 ⚠️（建议但不阻断）
- 报告中标注 `[待验证]`
- 说明"此判定基于逻辑推理而非实证，用户应自行判断"

### 为什么需要这个机制

审查角色的激励结构天然偏向"找过度复杂"，结论默认倾向"你想多了"。技术错误可以从代码直接证明，但"该不该做"是因果判断——不能用逻辑自洽替代实证验证。此机制确保推翻设计决策的成本与其影响匹配。

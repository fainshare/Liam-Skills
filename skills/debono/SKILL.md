# Debono 德伯诺回溯 · 太虚五转·归真

> *"The purpose of thinking is to abolish thinking."*
> — Edward de Bono
>
> 批判能找到漏洞，但找不到价值。归真是在批判的废墟里，捡回不该被埋没的东西。

## 概述

方案被批判管线打得很惨。但被否的不一定是错的，可能只是论证不完整、证据还没到、角度被忽略。批判有时下手太重，把有价值的东西连同不可靠的部分一起扔掉了。

这个 skill 为被质疑的命题构建最强存活论证：明确它在什么条件下仍然成立，从四个维度扫描被忽略的价值。存活条件，找到让命题成立的最小前提集合；被误伤的价值，识别批判中被一并否掉的合理内核；正向论证空间，从支持方向重新构建论证链；利益相关者关联，判断哪些人最关心这个命题、他们的诉求是什么。费曼（破妄）找漏洞，贝叶斯（叩实）验真伪，德伯诺（归真）构建支持。批判排除了不可靠的部分后，回废墟里找回被误杀的东西。

太虚五转第五转：归真（回溯）。散怀、澄源、叩实、破妄、归真，五转闭环。批判管线的终点，是回溯的起点。

## 定位

### 解决什么

Feynman 审查产出、Bayes 验证结论、Hegel 收敛认知，这三者形成了一条强大的批判管线。但批判管线有盲区：它擅长排除，不擅长保留。

Debono 补的是这块盲区：**给被否定的命题做逆向价值扫描**，不问"为什么错"，问"在什么条件下对"。

### 太虚五转中的位置

```
太虚一转：散怀  →  太虚二转：澄源  →  太虚三转：叩实  →  太虚四转：破妄  →  太虚五转：归真
 (osborn 发散)     (hegel 收敛)       (bayes 验证)        (feynman 审查)       (debono 回溯)
```

批判管线（澄→验→破）负责排除不可靠的东西，归真负责在排除之后回头看一眼，有没有被误杀的。

### 与其他 skill 的分工

| Skill | 方向 | 问什么 |
|-------|------|-------|
| feynman | 🔍 找漏洞 | 这份产出在骗自己什么？ |
| bayes | 📊 验真伪 | 这个结论经得起数据打脸吗？ |
| hegel | 🔄 收敛 | 不确定项收敛了吗？ |
| **debono** | 💡 找价值 | 被否掉的东西，有没有被忽略的价值？ |

Feynman + Debono 是认知的两种互补本能：怀疑与信任、批判与建设、破妄与归真。

## 构建方法论

构建存活论证时，从以下 5 种策略中选择最适用的 2-3 种组合使用。**禁止凭空编造正面理由**——每种策略都有明确的分析路径，沿路径走不通就换下一种。

| # | 策略 | 分析路径 | 适用场景 |
|---|------|----------|----------|
| S1 | **逆向利益相关方** | 列出所有利益相关方 → 找到被批判视角忽略的群体 → 分析该群体为何从中受益 | 方案被单一视角否定时 |
| S2 | **条件缩窄法** | 命题在「全场景」不成立 → 逐步缩窄适用条件 → 找到「窄但真实」的成立区间 | 命题被「过度泛化」反驳时 |
| S3 | **时间延迟价值** | 当前无价值 → 分析 6 个月/1 年/3 年后什么条件变化 → 找到价值涌现的时间窗口 | 命题被「当前不可行」否定时 |
| S4 | **类比复活法** | 在其他领域/行业/历史中找到类似命题成功的案例 → 分析成功条件是否可迁移 | 命题被认为「没人这么做过」时 |
| S5 | **反面代价法** | 如果不采纳此命题 → 分析会失去什么机会/承受什么代价 → 反向论证保留价值 | 命题被「不需要」否定时 |

**执行纪律**：
- 每种策略最多产出 1 个最强论点，不堆砌弱论点
- 走不通的策略诚实标注「此路径未发现存活论证」
- 最终论证中标注使用了哪些策略（可追溯）

### 乐观强度旋钮

根据命题被否定的严重程度，选择构建强度：

| 档位 | 名称 | 行为 |
|------|------|------|
| L1 | 结构化建设性（默认） | 选 2-3 种最适用策略，诚实标注走不通的路径 |
| L2 | 全力辩护 | 5 种策略全试，允许弱论点作为补充角度（仍禁止凭空编造） |

**选择规则**：默认 L1。当用户明确要求「全力辩护」「帮我想尽一切办法」或命题对业务有重大战略意义时升至 L2。

### 策略选择决策树

按否定类型推荐策略优先级：

| 否定类型 | 推荐优先级 | 默认组合 |
|----------|-----------|----------|
| 「视角单一」型否定 | S1 → S5 → S3 | S1+S5 |
| 「过度泛化」型否定 | S2 → S3 → S4 | S2+S3 |
| 「当前不可行」型否定 | S3 → S4 → S2 | S3+S4 |
| 「没人做过」型否定 | S4 → S1 → S5 | S4+S1 |
| 「不需要」型否定 | S5 → S1 → S2 | S5+S1 |
| **否定类型不明确** | S1 → S5 → S4 | **S1+S5**（默认） |

**使用方式**：先判断否定类型 → 按优先级选 2-3 种策略 → 否定类型不明确时使用默认组合 S1+S5。

→ 四维框架的理论根基详见 [references/methodology.md](references/methodology.md)

## 铁律

执行 debono 时，**禁止**在输出中出现以下内容：

- "但是"、"不过"、"然而"、"风险在于"、"潜在问题"
- 任何形式的风险提示、问题指出、"值得注意"的转折
- 隐性批判："这个方案虽然…但…"的句式

你是**辩护律师**，不是法官。法官的工作由 feynman/think-tool 承担。
唯一例外：在"存活条件"段落中，可以诚实说明"命题在什么条件下**不**成立"——这是边界标注，不是批判。

## 输入与输出

### 输入

```yaml
proposition: string | string[]   # 被质疑/被否定的命题（单条或列表）
context: string                  # 可选，命题的来源背景（如 hegel report 摘要）
```

### 输出

```markdown
## 最强存活论证
{为什么这个命题值得保留，在什么条件下它仍然成立}

## 存活条件
{列出命题成立所需的明确条件，不成立的情况同样诚实标注}

## 四维价值扫描
| 维度 | 评分 | 说明 |
|------|------|------|
| 用户价值 | {一句话} | {解决什么痛点、创造什么爽点} |
| 技术收益 | {一句话} | {技术债减少、复用性提升、架构简化} |
| 团队成长 | {一句话} | {能力建设、认知提升、协作改善} |
| 战略对齐 | {一句话} | {与上层目标/长期方向的一致性} |
```

## 执行流程

```
命题输入 → 构建存活论证 → 明确存活条件 → 四维价值扫描 → 输出 → [五转总结]
```

1. **接收命题**：单条或列表，带或不带上下文
2. **构建最强存活论证**：从 5 种构建策略中选择 2-3 种，沿分析路径生成论证
3. 🟡 **CHECKPOINT — 存活论证确认**：展示构建的存活论证摘要，询问用户：
   - 「以上存活论证方向是否正确？需要调整或补充角度吗？」
   - 用户确认后继续，用户修正则按反馈调整
4. **明确存活条件**：诚实标注——什么情况下成立、什么情况下不成立
5. **四维价值扫描**：从用户价值、技术收益、团队成长、战略对齐四个维度扫描被忽略的价值
6. **输出**：Markdown 段落，可直接嵌入 hegel 报告或其他文档
7. **🔒 五转综合总结（强制）**：当 debono 作为太虚五转管线的最后一步执行时，**必须**生成五转综合总结文档。详见下方「五转综合总结协议」。

## 五转综合总结协议（V2.2 新增）

当 debono 作为太虚五转管线（taixu-pipeline）的最后一步执行时，**必须**在 debono 输出之后，生成一份覆盖全部五转的综合总结文档。

### 触发条件

满足以下任一条件即触发：
- taixu-pipeline 编排器调用 debono 时传入 `--pipeline-context` 参数
- 当前会话中存在前四转的产出物文件（osborn-pool + hegel-report + bayes-report + feynman-report）
- 用户显式传入 `--summary` 参数

**不触发**：debono 独立调用（非五转管线上下文）时不执行此步骤。

### 执行步骤

1. **收集前四转产出物**：读取 osborn-pool、hegel-report、bayes-report、feynman-report 的路径和内容摘要
2. **读取本次 debono 产出**：存活论证 + 四维价值扫描
3. **生成综合总结**：按下方模板写入 `docs/reports/{topic-slug}-taixu-summary-{date}.md`

### 设计约束

- **必须综合全部五转**：不是只总结 debono，是从调研/问题定义到 debono 的完整叙事
- **必须包含"元认知收获"**：对本次五转流程本身的反思（参见 Feynman 发现的流程级偏差）
- **必须写入文件**：产出物落盘到 `docs/reports/`，不只在对话中输出
- **字数控制**：综合总结 1500-3000 字，不超过任何单转的报告长度

## 边界条件

| 场景 | 处理 |
|------|------|
| 命题确实无价值 | 诚实输出「未发现显著存活价值」，不强行建设 |
| 命题超出四维框架 | 四维是默认框架，特定场景可扩展维度 |
| 单命题 vs 批量 | 接口同时支持两种模式，批量时逐命题输出 |
| 输出过长 | 每条命题的存活论证控制在 300 字内 |
| 输入为空或纯情绪 | 回复「请提供具体命题（可验证的陈述或方案），纯情绪表达无法做价值回溯」 |
| 非命题类文本（如代码片段） | 回复「debono 处理的是命题/方案/决策，代码/数据类输入请用 feynman 审查」 |
| 5 种策略全部走不通 | 诚实输出「5 种构建策略均未发现存活论证，该命题在当前上下文中不具备保留价值」 |

## 反模式

| 坑 | 表现 | 怎么防 |
|----|------|--------|
| 强行乐观 | 明明没价值还硬编正面角度 | 诚实第一，无价值就说无价值 |
| 替代批判 | 用 debono 逃避 feynman 审查 | debono 是补充不是替代 |
| 过度建设 | 把 1/10 的价值放大成 9/10 | 四维评分各限 1 句话 |
| 忽略存活条件 | 只说"有价值"不说"什么时候有价值" | 存活条件是强制输出项 |

## 跨 Skill 触发协议

### hegel → debono（S7 黄帽对冲）

当 hegel 收敛报告满足以下条件时，建议用户触发 debono：

**触发条件**：hegel 报告中 >70% 的初始发现被标记为 `dismissed` 或 `excluded`

**输入提取规则**（从 hegel 报告中提取）：

```yaml
proposition: <hegel 报告中被 excluded/dismissed 的 findings 列表>
context: <hegel 报告的 scope + convergence summary>
```

**输出嵌入**：debono 输出可直接追加到 hegel 报告末尾的 `## 黄帽对冲（debono 价值回溯）` 段落。

### feynman ↔ debono（双方论证）

对同一命题分别调用 feynman 和 debono，获得正反两面：
- feynman 输出：自欺点 + 未验证假设
- debono 输出：存活论证 + 四维价值
- 用户综合两面做最终决策

### 沉默触发器

在对话过程中，当检测到以下信号时，**建议**用户触发 debono（建议模式，非自动执行）：

| 信号 | 识别方式 | 建议话术 |
|------|----------|----------|
| hegel dismiss 比例 >70% | 从 hegel 报告中读取 dismiss 统计 | 「收敛结果中大部分发现被排除了，要不要跑一轮 debono 看看有没有被误杀的？」 |
| 用户犹豫 | 用户表达「觉得有道理但不确定」「被否了但还想再看看」 | 「听起来你对这个结论有保留，要不要用 debono 做一轮价值回溯？」 |
| feynman 后直觉反抗 | 用户看完 feynman 审查后表达「说得对但我不想放弃」 | 「费曼找到了漏洞，但如果你觉得核心想法仍有价值，debono 可以帮你构建存活论证」 |

## 设计决策

| # | 决策 | 理由 |
|---|------|------|
| D1 | 独立 skill 而非 hegel 内嵌 | 可被任何环节独立调用，不依赖 Hegel 管线 |
| D2 | V1 固定"结构化建设性"模式 | 降低复杂度，V2 已通过乐观强度旋钮实现（D5） |
| D3 | 输出格式兼容 hegel target_text | 确保可直接嵌入 hegel S7 报告 |
| D4 | 四维框架为默认，非硬性约束 | 特定领域可扩展维度，保持灵活性 |
| D5 | 乐观强度 2 档而非 3 档 | 五转审查发现 L2/L3 边界不清（"最强+次强" vs "所有策略必须产出"实际执行中难以区分），简化为 L1 默认 + L2 全力辩护 |
| D6 | 策略推荐优先级 + 默认组合 | 五转审查发现单一映射过于简化，改为按否定类型推荐优先级，不明确时默认 S1+S5（实测适用面最广） |
| D7 | 沉默触发器为建议模式 | 自动执行可能导致用户被强行拉入 debono 流程，建议模式保留用户决策权 |
| D8 | 五转总结放在 debono 而非 pipeline | debono 是五转最后一步，天然拥有全部前序产出物的上下文；放在 pipeline 会新增步骤，增加编排复杂度。debono 独立调用时不触发总结，不影响独立性 |
